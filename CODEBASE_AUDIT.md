## Findings summary

### Duplication

Three structural duplications stand out:

1. **Binary node layout knowledge is duplicated three ways** between `crates/satteri-ast/src/rebuild.rs:390-494` (`remap_*_string_refs`), `crates/satteri-ast/src/walk.rs:162-507` (`serialize_*_node_inline`), and the per-node `*Data::to_bytes/from_bytes` impls in `crates/satteri-ast/src/mdast/codec.rs:20-281`. They all describe the same schema and must stay in lockstep. A `node_layout` module returning `&'static [StringRefField]` per `(kind, node_type)` would drive all three.
2. **`rebuild.rs` has four near-identical copy-emit functions** â€” `copy_node` (133-261), `emit_subtree_node` (281-321), `emit_subtree_with_original_children` (324-371), `copy_node_raw` (571-607) â€” sharing prologue (open_node_raw â†’ set_node_data â†’ set_position) and epilogue (children loop + close_node).
3. **HAST and MDAST visitor runtimes are copy-pasted** in TypeScript: `packages/satteri/src/hast/hast-visitor.ts:691-792` is byte-identical (apart from the context type) to `packages/satteri/src/mdast/mdast-visitor.ts:735-792` for the dispatch loop. AGENTS.md says hast/mdast node *types* mirror each other intentionally; the runtime mechanics needn't.

Secondary repeats: ~50Ă— `getString(ref.offset, ref.len)` ceremony across the two readers; 20+ identical `handle.lock().map_err(...)` blocks in the NAPI binding; an existing `lazyProp` helper in `packages/satteri/src/lazy-props.ts` that ~10 sites in the visitors/materializers reinvent inline.

### Performance pitfalls (static reading, not measured)

Ranked by likely flamegraph impact:

1. **`DataMap` allocates a `String` per get/set/has/remove** â€” `crates/satteri-plugin-api/src/data.rs:52,56,60,64`. Called per-node by plugin code, so this likely sits high on a real flamegraph.
2. **`[Vec<â€¦>; 256]` allocated every walk call** â€” `crates/satteri-ast/src/walk.rs:71` builds 256 vectors regardless of how many node types the caller subscribes to, plus a `Vec<String>` `tag_filter` linear-scanned per node.
3. **HTML escape uses `.chars()` over ASCII-heavy attrs**, and `properties.rs:63` heap-allocates per camelCase attribute (`crates/satteri-ast/src/hast/render.rs:30`, `crates/satteri-ast/src/hast/properties.rs:63`).
4. **`convert.rs` allocates per-element** â€” `merged_h_props` builds `Vec<(String, â€¦)>` even when no overrides, plus repeated `format!("language-{}", â€¦)` / `format!("user-content-fn-{}", â€¦)` on common nodes (lines 138, 1025, 1126, 1438-1444, 1861, 1946, 1980-1987).
5. **`apply_patches` clones the full source per command** â€” `crates/satteri-arena/src/arena.rs:180-193` does `arena.source().to_string()`; `crates/satteri-ast/src/rebuild.rs:110,218` does per-node `node_data.to_vec()`. An `Arc<str>` source would make this O(1).

The Rust hot paths already use `rustc_hash::FxHashMap` pervasively, and the TS hot paths already cache `TextDecoder` / `TextEncoder`. No stray `println!` / `dbg!` calls remain.

### Code quality

The repo is unusually clean for its size: **zero `#[allow(dead_code)]` annotations**, zero commented-out code blocks, no swallowed catch handlers of consequence, and the panics that do exist are invariant-protected. The real issues cluster around:

- **Parse errors are silently dropped at the public NAPI entry points** â€” `crates/satteri/src/lib.rs:12`, `crates/satteri-napi-binding/src/lib.rs:242,288,500` all do `let (arena, _) = â€¦parse(â€¦)`, while the MDX siblings at `:302,:513` do check `mdx_errors.first()`. The policy needs to be uniform.
- **Two enormous functions in MDX JSX rewriting** â€” `process_sub_expressions` (731 lines) and `process_statement_jsx` (667 lines) at `crates/satteri-mdxjs-rs/src/oxc_util_build_jsx.rs:175,916` thread 11 parameters through deep recursion. A `JsxBuildCtx<'a>` would clean it up. `convert_node` at `crates/satteri-ast/src/convert.rs:948` (~640 lines) is a flat match that wants per-arm helpers.
- **`compile_handle` doc lies** â€” `crates/satteri-napi-binding/src/lib.rs:567-582` documents "Does not consume the handle" but `simplify_plain_mdx_nodes` mutates in place.
- **`@ts-nocheck` on `packages/satteri/src/binding.browser.ts:1`** erases types across the WASM build, defeating strict typing elsewhere.
- **`CommandError` rolls `Display`/`Error` by hand** at `crates/satteri-ast/src/commands.rs:91-136`, even though `CONTRIBUTING.md` calls out `thiserror` for typed enums.
- **Magic node-type numbers** in `crates/satteri-ast/src/rebuild.rs:392,420,456,483` (`100 | 101`, `30..=32`) bypass the typed enums and silently break if IDs are renumbered.
